我想要编写一个CMake 项目生成器名字叫fast build generate ，简称fb-gen,扫描 C/C++ 项目目录结构，自动生成可编译的多模块 CMake 配置.
核心理念:
1. 发现即结构：每个包含至少一个 .c/.cpp 源文件的子目录，都自动成为一个独立的 CMake 子模块（add_subdirectory）。
2. 零配置假定：工具不依赖额外的配置文件，通过扫描目录结构和文件内容智能推断模块边界与依赖关系。
3. 内容感知能力：利用 fff-search 的内容搜索能力，通过扫描 #include 语句推断头文件归属，建立模块间的依赖关系（可选增强特性）。通过fff-search 的内容搜索能力,读取用户对CMakeLists配置文件的修改，更新配置缓存。
4. 增量友好：当文件结构发生变化时，可快速重新生成配置，适合活跃开发的代码仓库。
5. 用户友好型设计，提供cli命令，方便用户直接介入程序运行逻辑（项目初始化，手动同步配置，依赖查询，基于cmake的第三方库管理）


架构分层设计：
1 CLI 交互层 : clap / 用户命令解析
2. 编排与协调层:工作流串联、进度报告
3. 核心业务逻辑层    :模块发现、依赖分析、CMake生成,项目元数据管理，第三方包管理（不参与包的引入）

4. 外部依赖/工具层:文件系统、fff_search、stdlib,clap,cammad,tera模板渲染。
注：各层职责明确，上层可调用下层接口，下层不感知上层存在。

各层核心功能模块功能详细分配：
第一层：外部依赖/工具层

模块 职责 关键依赖
文件系统适配器 抽象底层文件操作，便于测试 std::fs
fff_search 封装 调用 fff_search 进行高性能目录扫描和内容检索 fff_search crate

第二层：fff-search 抽象层

这是本工具与常规目录扫描工具的核心差异所在。抽象层的设计目标是：将 fff_search 的高性能能力封装为便于上层消费的语义化接口。

接口 输入 输出 说明
scan_source_files() 根目录路径 源文件列表（含路径、类型、大小） 利用 fff_search 的文件索引能力快速遍历
scan_subdirs() 根目录路径 子目录树结构 获取目录层级关系
scan_includes() 源文件路径 该文件引用的头文件列表 利用 grep 模块内容搜索能力解析 #include
watch_changes() 根目录路径 文件变更事件流 利用 fff_search 的文件监控能力支持增量生成

💡 关键设计决策：fff_search 的核心优势在于其“常驻内存”特性。本工具在首次扫描时构建内存索引，后续增量操作只需极短时间即可完成，非常适合在 CI 环境或编辑器插件中高频使用。
第三层：核心业务逻辑层

这一层实现生成 CMakeLists.txt 的核心算法：

模块 输入 输出 核心逻辑
模块发现器 项目根目录 模块列表（每个模块：路径、源文件列表、头文件列表） 递归扫描：每个包含源文件的目录 → 一个 CMake 模块
依赖分析器 模块列表 模块间依赖图 扫描每个模块的 #include，匹配其他模块的头文件路径
CMake 生成器 模块列表、依赖图 各目录的 CMakeLists.txt 文件 根据模块类型生成 add_executable/add_library，插入依赖关系
配置推断器 模块上下文 编译选项（C++ 标准、宏定义等） 可选：通过扫描源文件特征推断

模块发现的特殊规则：

· 根模块：根目录生成顶层 CMakeLists.txt，负责 include_directories 和 add_subdirectory。
· 叶子模块：每个子目录独立生成 CMakeLists.txt，定义自己的 target。
· 忽略规则：自动忽略 build/、.git/、third_party/ 等常见目录。

依赖分析策略（增强特性）：

策略 说明 适用场景
路径推断 #include "core/foo.h" → 尝试匹配 core/ 子目录 常规项目
内容匹配 利用 fff_search 扫描整个项目，查找定义该头文件的目录 头文件不在标准路径时
启发式规则 头文件名与模块名相近时自动关联 松散结构的项目

组件 职责

第五层：CLI 交互层

命令 功能
fb init 初始扫描并生成完整的 CMakeLists.txt 结构
fb sync 增量更新：仅重新生成发生变化的模块
fb check 校验模式：检查现有 CMakeLists.txt 与当前目录结构的一致性,输出diff
fb validate 生成后自动调用 cmake --build 验证配置正确性
fb run 调用Cmake构建，ninja或make编译，
支持的 CLI 选项：

· --root <path> — 项目根目录（默认当前目录）
· --exclude <dirs> — 排除的目录列表
· --lang <c/cpp> — 仅处理特定语言的文件,指定c/cpp标准
· --no-deps — 跳过依赖分析，仅生成模块骨架
· --output <dir> — 指定输出目录（用于预览模式）默认build/
· --watch=enable/disable — 启动文件监控，自动增量更新，默认启动


CMake位置生成流程：
文件扫描  │ -> │ 模块分组  │ -> │ 依赖分析  │ -> │ CMake生成 │
阶段 输入 输出 说明
文件扫描 根目录路径 Vec<SourceFile> 利用 fff_search 获取所有 C/C++ 源文件和头文件,同时根据用户对CMakeLists.txt的修改更新项目模型
模块分组 Vec<SourceFile> Vec<CMakeModule> 按文件所在目录分组，每个目录形成一个潜在模块
依赖分析 Vec<CMakeModule> Vec<CMakeModule>（补充依赖） 分析每个模块源文件中的 #include 语句，匹配到其他模块
CMake生成 Vec<CMakeModule> 各目录下的 CMakeLists.txt 将模块模型渲染为实际的 CMake 配置



增量更新策略

当文件系统发生变化时，增量更新的核心逻辑：

1. 变更检测：利用 fff_search 的文件监控能力（watch_changes 接口）获取变动的文件列表。
2. 影响范围计算：
   · 新增/删除源文件 → 仅影响所在模块
   · 新增/删除目录 → 可能新增/删除整个模块
   · 修改 #include 语句 → 可能影响依赖关系
3. 局部重新生成：仅对受影响的模块重新生成 CMakeLists.txt。
4. 依赖图更新：若依赖关系发生变化，重新分析并更新顶层配置。

输出规格：生成的 CMakeLists.txt 结构

顶层 CMakeLists.txt（根目录）

```cmake
cmake_minimum_required(VERSION 3.10)
project(MyProject VERSION 0.1.0 LANGUAGES CXX)

# 全局头文件搜索路径
include_directories(${CMAKE_CURRENT_SOURCE_DIR}/include)

# 递归添加所有子模块
add_subdirectory(core)
add_subdirectory(utils)
add_subdirectory(network)
add_subdirectory(app)
```

子模块 CMakeLists.txt（以 core/ 为例）

```cmake
project(core)

# 收集当前目录下的所有源文件
file(GLOB_RECURSE SOURCES "src/*.cpp" "src/*.c")
file(GLOB_RECURSE HEADERS "include/*.h" "include/*.hpp")

# 生成库目标
add_library(core STATIC ${SOURCES} ${HEADERS})

# 依赖其他模块
target_link_libraries(core PRIVATE utils)

# 设置编译选项
target_compile_features(core PUBLIC cxx_std_17)
target_include_directories(core PUBLIC ${CMAKE_CURRENT_SOURCE_DIR}/include)
```

场景 行为
目标文件已存在 对比现有内容与生成内容，若不一致则生成 CMakeLists.txt.new 并给出 diff 提示
用户自定义内容 支持在模块中保留用户手动添加的编译选项（通过标记块 # USER_START / # USER_END）
