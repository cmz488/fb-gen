# 工具链模板优化 & 自动检测 — 设计规范

## 概述

参考 STM32CubeMX 工具链模板改进 `toolchain.cmake` 生成质量，新增项目级自动检测（根目录 `.ld` 链接器脚本、用户自定义 CMakeLists.txt），确保所有输出中的依赖声明按拓扑顺序排列。

---

## 变更 1：工具链模板改进（参考 STM32CubeMX）

### `CMAKE_ASM_FLAGS` 修复

**之前**（错误）：
```cmake
set(CMAKE_ASM_FLAGS "${CMAKE_ASM_FLAGS} ${TARGET_FLAGS} -x assembler-with-cpp -MMD -MP")
```
`CMAKE_ASM_FLAGS` 初始为空，因此仅获得 `TARGET_FLAGS`，缺少 `-Wall -fdata-sections` 等。

**之后**（参考 CubeMX）：
```cmake
set(CMAKE_ASM_FLAGS "${CMAKE_C_FLAGS} -x assembler-with-cpp -MMD -MP")
```
`CMAKE_C_FLAGS` 包含 `TARGET_FLAGS` + `-Wall -fdata-sections -ffunction-sections -fstack-usage`，所有内容均传递至汇编编译器。

### 对齐格式 & 注释

参考 CubeMX 的视觉风格：
```cmake
set(CMAKE_SYSTEM_NAME               Generic)
set(CMAKE_SYSTEM_PROCESSOR          arm)

# Some default GCC settings
# arm-none-eabi- must be part of path environment
set(TOOLCHAIN_PREFIX                arm-none-eabi-)
```

## 变更 2：自动检测根目录链接器脚本

### 行为

- 在生成过程中，扫描 `config.root`（仅一级深度，不递归）以查找 `.ld` 文件
- **发现 1 个 `.ld`**：将其作为 `-T "${CMAKE_SOURCE_DIR}/<name>.ld"` 自动添加到 `CMAKE_EXE_LINKER_FLAGS`
- **发现 0 个**：不添加 `-T` 标志（无错误）
- **发现 2+ 个**：不添加 `-T` 标志（歧义；用户必须通过 `USER_START`/`USER_END` 自行添加）

### 实现

- 在 `CMakeGenerator::generate()` 中新增 `detect_linker_scripts(root: &Path) -> Vec<PathBuf>`
- 将其传入 `render_embedded_toolchain()`

### 模型变更

从 `ToolchainConfig` 中**移除** `linker_script: String` 字段。

**受影响的位置**：
- `src/models/project.rs`：从 `ToolchainConfig` 中移除 `linker_script` 字段
- `src/orchestration/query.rs`：移除链接器脚本提示
- `src/core/generator.rs`：将 `detect_linker_scripts()` 的结果注入模板
- `tests/integration.rs`：移除使用 `linker_script` 的测试配置

## 变更 3：检测用户自定义的 CMakeLists.txt

### 行为

`scan_and_discover()` 新增一个步骤：
1. 遍历项目子目录（排除 `exclude_dirs`）
2. 查找现有的 `CMakeLists.txt` 文件
3. 读取每个文件：检查 `# Generated by fb-gen` 头部
4. 没有 fb-gen 头部的文件 → **用户模块**：记录其目录路径

### 用户模块处理

- 用户在根 CMakeLists.txt 的 `USER_START` 块中列出 `add_subdirectory`，以**保护其不被覆盖**
- 用户在用户模块的 `add_subdirectory` 声明之后列出生成模块
- fb-gen **不会**为用户模块目录生成新的 `CMakeLists.txt`

### 根 CMakeLists.txt 输出片段

```cmake
# ── User customisations ──────────────────────────────────────────────
# USER_START
# 用户自定义的 CMake 模块（非 fb-gen）：
add_subdirectory(Drivers)
add_subdirectory(Middlewares)
# USER_END

# ── Sub-modules ──────────────────────────────────────────────────────
add_subdirectory(Core)
add_subdirectory(App)
```

### 实现

- 在 `src/scanner/fff_wrapper.rs` 中新增 `scan_user_cmake_files(root: &Path, exclude_dirs: &[String]) -> Vec<PathBuf>`
- 在 `scan_and_discover()` 中调用以填充 `Vec<PathBuf>`（需要传递给生成器）
- 修改 `render_root()` 以接收 `user_modules: &[PathBuf]`
- 调整 `USER_START`/`USER_END` 块以包含预先填充的用户模块指令

## 变更 4：模块 `target_link_libraries` 的拓扑顺序

### 行为

模块 CMakeLists.txt 中的依赖项目前为任意顺序（图的插入顺序）。应改为按拓扑顺序列出：首先列出库依赖项，然后列出可执行文件。这可以防止因链接顺序导致的 "undefined reference" 错误。

### 实现

在 `render_module()` 中，接收依赖项列表并按拓扑顺序排序：
```rust
// 按拓扑顺序排序依赖项：依赖项在前，依赖者在后
let ordered_modules = graph.topological_order().unwrap_or_default();
let mut sorted_deps = deps.clone();
sorted_deps.sort_by_key(|(name, _)| {
    ordered_modules.iter().position(|n| n == name).unwrap_or(usize::MAX)
});
```
