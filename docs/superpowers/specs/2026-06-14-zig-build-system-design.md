# Zig Build System Integration — Design Spec

## Goal

Add Zig as an alternative build system alongside CMake. Users choose at `init` time; the project config persists the choice; all subsequent commands dispatch to the correct generator.

## Architecture

### Two independent generators

```
src/core/
  cmake_generator.rs    ← existing, unchanged
  zig_generator.rs      ← NEW, parallel module
```

Both consume the same inputs — `ProjectConfig`, `Vec<CMakeModule>`, `DependencyGraph` — produced by the shared Scanner → Discoverer → Analyzer pipeline.

### Module boundaries

```
CLI (commands.rs)
  ├───────── Orchestration (workflow, cache, watcher, reporter)
  │              │
  │              └── Core   ── ModuleDiscoverer      (shared)
  │                         ── DependencyAnalyzer    (shared)
  │                         ── CMakeGenerator        (CMake path)
  │                         ── ZigGenerator          (Zig path)    ← NEW
  │              │
  │              └──  Scanner ── FffScanner           (shared)
  │                         ── fs_adapter            (shared)
  │
  └── Dispatch:
        config.build_system → CMakeGenerator or ZigGenerator
```

### Dispatch point

In `commands.rs`, after scan→discover→analyze, the generate phase branches:

```rust
match config.build_system {
    BuildSystem::CMake => {
        let gen = CMakeGenerator::new(&config)?;
        gen.generate(&modules, &graph, force, &user_modules)?;
    }
    BuildSystem::Zig => {
        let gen = ZigGenerator::new(&config)?;
        gen.generate(&modules, &graph)?;
    }
}
```

Orchestration, incremental sync, and file watching are unchanged — `do_incremental_sync` calls the same shared pipeline, then dispatches to the appropriate generator at the end.

## New types

### `BuildSystem` enum (`src/models/project.rs`)

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub enum BuildSystem {
    CMake,   // default
    Zig,
}
```

### `ProjectConfig` addition

```rust
pub struct ProjectConfig {
    // ... existing fields ...
    pub build_system: BuildSystem,
}
```

### `ZigGenerator` (`src/core/zig_generator.rs`)

```rust
pub struct ZigGenerator {
    config: ProjectConfig,
}

impl ZigGenerator {
    pub fn new(config: &ProjectConfig) -> FbGenResult<Self>;
    /// Generate `build.zig` at the project root.
    pub fn generate(
        &self,
        modules: &[CMakeModule],
        graph: &DependencyGraph,
    ) -> FbGenResult<()>;
}
```

## Target architecture mapping

| fb-gen `TargetArch` | Zig `cpu_arch` | `os_tag` | CPU model example |
|----------------------|----------------|----------|-------------------|
| `X86_64` | `.x86_64` | native | — |
| `X86` | `.x86` | native | — |
| `ARM32` | `.arm` | `.freestanding` | `.cortex_a7` |
| `ARM64` | `.aarch64` | `.freestanding` | `.cortex_a53` |
| `NoneEabi` | `.thumb` | `.freestanding` | `.cortex_m4` |
| `RISCV64` | `.riscv64` | `.freestanding` | `.baseline_rv64` |
| `RISCV32` | `.riscv32` | `.freestanding` | `.baseline_rv32` |
| `Xtensa` | `.xtensa` | `.freestanding` | `.esp32s3` |
| `WASM` | `.wasm32` | `.freestanding` | `.baseline` |
| `Custom(label)` | user input | `.freestanding` | user input |

The CPU model is collected during `init` and stored in `ProjectConfig.toolchain.cpu`.

## Dependency mapping

`DependencyAnalyzer` output (module-level edges with PUBLIC/PRIVATE types) maps to Zig build directives:

| Analyzer edge | CMake expression | Zig expression |
|---------------|-----------------|----------------|
| A dep B (PUBLIC) | `target_link_libraries(A PUBLIC B)` | `A.addIncludePath(B_dir); A.linkLibrary(B)` |
| A dep B (PRIVATE) | `target_link_libraries(A PRIVATE B)` | Same as above, no transitive propagation needed |

Zig handles file-level incremental compilation internally — fb-gen does not emit per-file dependency rules.

## Generated `build.zig` structure

```zig
// Auto-generated by fb-gen — do not edit manually.
const std = @import("std");

pub fn build(b: *std.Build) void {
    const target = b.resolveTargetQuery(.{
        .cpu_arch = .xtensa,
        .cpu_model = .{ .explicit = &std.Target.xtensa.cpu.esp32s3 },
        .os_tag = .freestanding,
    });
    const optimize = .ReleaseSmall;

    // ── Modules ────────────────────────────────────────────────
    const mod_core = b.addStaticLibrary(.{
        .name = "core", .target = target, .optimize = optimize,
    });
    mod_core.addCSourceFiles(.{ .files = &.{ "core/foo.c", "core/bar.c" }, .flags = &.{} });
    mod_core.addIncludePath(.{ .path = "core" });

    const mod_src = b.addExecutable(.{
        .name = "app", .target = target, .optimize = optimize,
    });
    mod_src.addCSourceFiles(.{ .files = &.{ "src/main.cpp" }, .flags = &.{} });
    mod_src.addIncludePath(.{ .path = "src" });

    // ── Dependencies ────────────────────────────────────────────
    mod_src.addIncludePath(.{ .path = "core" });
    mod_src.linkLibrary(mod_core);

    // ── PIO framework (ESP32 only) ──────────────────────────────
    // Include dirs and source files resolved from
    // ~/.platformio/packages/framework-arduinoespressif32/
    mod_src.addIncludePath(.{ .path = "/home/.../cores/esp32" });
    // ...

    b.installArtifact(mod_src);
}
```

## `init` flow adjustments

When the user selects Zig:

1. **Target architecture** — same as CMake path
2. **CPU model** — prompt with architecture-appropriate candidates
3. **Optimization** — `Debug` / `ReleaseSafe` / `ReleaseSmall` / `ReleaseFast`
4. **Skip BuildBackend** — Zig is its own build system; no Ninja/Make question
5. **Skip compiler selection** — Zig is both; no GCC/Clang question
6. **PIO framework** — when target is Xtensa, ask whether to integrate

## CLI commands

```
fb-gen init                  ← interactive, asks build system
fb-gen init --build=zig      ← non-interactive: pick Zig
fb-gen sync                  ← reads config.build_system, dispatches
fb-gen run                   ← zig build (Zig path) or cmake --build (CMake path)
fb-gen check                 ← same dispatch
fb-gen run --lsp             ← zig generates compile_commands.json natively
```

## Zig availability

- Zig must be installed system-wide (`zig` on PATH)
- `fb-gen init --build=zig` checks: `zig version` succeeds, otherwise errors with guidance
- No auto-download; user responsibility

## What does NOT change

- `FffScanner` / `ScanOptions` — file discovery
- `ModuleDiscoverer` — grouping files into `CMakeModule` structs
- `DependencyAnalyzer` — `#include` parsing and `DependencyGraph` construction
- `MetaCache` — incremental sync checksums
- `FileWatcher` — change detection for `--watch`
- `Reporter` — progress output
- `UserQuery` — interactive prompt helpers

## Verification

1. `cargo build` — clean compilation
2. `cargo test --lib` + `cargo test --test integration` — all existing tests pass (zig is additive)
3. Manual: `fb-gen init --build=zig` on a simple C project → verify `build.zig` is generated
4. Manual: `zig build` on the generated project → compiles successfully
5. Manual: `fb-gen sync` → incremental update of build.zig
